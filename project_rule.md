# FFusion Media 项目开发规则手册

## 1. 项目基础信息
- 项目名称：FFusion Media
- 项目定位：基于FFmpeg与Python开发的跨平台本地媒体处理工具箱，为FFmpeg命令行工具提供可视化、低门槛的GUI封装，支持视频剪切、格式转换、音频处理、图像转换等核心功能，预留插件化扩展能力，支持后续新增功能。
- 目标用户：普通个人用户，无需掌握FFmpeg命令，即可完成常用媒体处理操作
- 开源协议：MIT
- 操作系统：Windows11
## 2. 技术栈强制规范
### 2.1 核心技术栈（固定不可修改）
依赖包等一定要安装到venv虚拟库中
| 模块 | 技术选型 | 版本要求 | 用途说明 |
|------|----------|----------|----------|
| 开发语言 | Python | 3.10+ | 全项目开发语言，必须兼容Python 3.10及以上稳定版本 |
| 核心引擎 | FFmpeg | 6.0+ 静态编译版 | 媒体处理核心，项目内置对应平台的静态二进制文件，无需用户手动安装 |
| GUI框架 | PySide6 | 6.7.2+ | 跨平台GUI开发，优先选用PySide6（LGPL协议，商用无风险），禁止使用PyQt6、Tkinter等其他框架 |
| 异步处理 | QThread | 随PySide6 | 媒体处理任务异步执行，防止GUI卡顿，禁止使用多进程、threading等其他异步方案 |
| 打包工具 | PyInstaller | 6.0+ | 跨平台可执行文件打包，必须适配Windows、macOS、Linux三大平台 |
| 配置管理 | pydantic-settings | 2.0+ | 用户配置管理，禁止硬编码配置项 |
| 日志系统 | Python logging | 标准库 | 全链路日志记录，禁止使用print输出运行信息 |

### 2.2 禁止使用的第三方库
- 禁止使用ffmpeg-python、PyAV等二次封装FFmpeg的库，必须使用Python标准库subprocess直接调用FFmpeg原生命令，保证可控性与兼容性
- 禁止使用未在本规则中指定的第三方库，如需新增必须提前申请，仅允许添加纯Python实现、无系统依赖的轻量库

## 3. 项目架构与目录结构规范
### 3.1 核心架构设计
采用**三层解耦架构**，严格分离界面层、业务逻辑层、核心引擎层，禁止跨层调用，保障可扩展性：
1.  核心引擎层：负责FFmpeg命令封装、执行、进度解析、媒体信息读取，与业务完全解耦
2.  业务功能层：基于核心引擎封装独立的功能模块，采用插件化设计，每个功能为独立可插拔模块
3.  GUI界面层：负责用户交互与界面渲染，仅与业务功能层对接，禁止直接调用FFmpeg命令

## 4. 代码编写强制规范
1.  编码规范：严格遵循PEP8规范，所有代码必须添加类型注解，禁止无类型的函数参数与返回值
2.  命名规范：
    - 类名采用大驼峰命名法（如FFmpegEngine）
    - 函数、变量、文件名采用蛇形命名法（如build_ffmpeg_command）
    - 常量采用全大写蛇形命名法（如DEFAULT_OUTPUT_PATH）
3.  异常处理规范：
    - 所有可能出错的操作必须做异常捕获，禁止裸写业务逻辑
    - 必须使用exceptions.py中定义的自定义异常，禁止直接抛出通用Exception
    - 所有异常必须写入日志，同时给用户提供友好的中文提示，禁止直接向界面抛出堆栈信息
4.  安全规范：
    - 所有用户输入的参数、文件名、路径必须做严格的转义与合法性校验，防止FFmpeg命令注入攻击
    - 禁止使用shell=True执行subprocess命令，必须使用列表传参模式
    - 所有文件操作必须做权限校验，防止越权读写
5.  跨平台兼容规范：
    - 所有路径处理必须使用pathlib标准库，禁止硬编码路径分隔符（/或\）
    - 自动识别当前操作系统，适配Windows、macOS、Linux的路径、权限、二进制文件差异
    - 禁止使用平台专属的API，如需使用必须做兼容降级处理
6.  可扩展规范：
    - 所有功能模块必须继承base.py中的BaseFunction基类，实现固定的接口方法
    - 核心引擎层禁止修改对外接口，如需扩展必须新增方法，不得修改原有方法的入参与返回值
    - 界面层与业务层解耦，新增功能模块无需修改主窗口代码，自动注册到侧边栏导航

## 5. 功能模块开发规范
所有功能模块必须遵循统一的开发标准，保障插件化可扩展：
1.  每个功能模块为独立的.py文件，存放于src/modules/目录下
2.  必须继承BaseFunction基类，实现以下核心属性与方法：
    - 基础属性：name（功能名称）、description（功能描述）、icon（功能图标）、params（参数字典）
    - 核心方法：validate_params()（参数校验）、build_command()（构建FFmpeg命令）、execute()（执行功能）、cancel()（取消执行）
3.  每个功能模块必须独立完成参数校验、命令构建、异常处理，不依赖其他功能模块
4.  所有功能必须支持批量处理、实时进度回调、执行状态反馈
5.  新增功能仅需在modules目录下新增对应模块文件，无需修改核心引擎与主窗口代码

## 6. 日志与配置规范
1.  日志规范：
    - 统一使用utils/logger.py中初始化的logger实例，禁止单独初始化logging
    - 日志分为DEBUG、INFO、WARNING、ERROR四个级别，FFmpeg执行命令、报错信息必须全量记录
    - 日志文件自动滚动存储，存放于用户目录下的.ffusion_media/logs文件夹，禁止存放于程序目录
    - 日志文件必须包含时间、级别、模块、行号、详细信息，方便问题排查
2.  配置规范：
    - 所有用户配置通过utils/config_manager.py统一读写，禁止直接读写配置文件
    - 配置文件存放于用户目录下的.ffusion_media/config.ini，禁止存放于程序目录
    - 必须提供配置默认值，即使配置文件损坏/丢失，程序也可正常启动运行
    - 配置项必须分类管理，分为基础设置、FFmpeg设置、界面设置、高级设置

## 7. 交付物规范
1.  所有交付的代码必须可直接运行，无语法错误、无依赖缺失
2.  每个模块必须配套对应的单元测试代码，存放于tests/目录
3.  每个阶段必须交付完整的文档，包括开发说明、自测报告、使用说明
4.  最终交付必须包含三大平台的打包好的可执行文件，无需用户安装任何依赖即可运行
